# PTRSS Sync Rules

这份文件用来明确区分：

- **仓库版**：GitHub 里的源码仓库
- **现网版**：当前正在运行的 PTRSS 实例

避免以后出现“我只是想整理仓库，结果把现网改了”这种事。

---

## 一、默认原则

### 默认只动仓库版，不动现网版

如果用户只是说：

- 整理一下
- 顺手同步 GitHub
- 改 README
- 清理目录
- 做成能进仓库的版本

默认理解为：

- 只改 **仓库版**
- 不改 **现网版**
- 不重启现网容器
- 不改真实配置

---

## 二、哪些改动属于“只改仓库版”

以下内容默认只进入仓库，不上线现网：

- README 调整
- `.gitignore` 调整
- 示例配置文件更新
- 目录重构
- 代码注释整理
- 脱敏处理
- 备份文件清理
- 仓库门面优化

这些改动不会直接影响当前运行中的 PTRSS。

---

## 三、哪些改动属于“可能影响现网”

以下内容一旦要真正上线，就可能影响现网：

- `app/app.py` 业务逻辑修改
- Web 页面行为修改
- Telegram 推送逻辑修改
- 下载器提交流程修改
- 轮询逻辑修改
- 配置读取 / 保存逻辑修改

这类改动即使先提交进仓库，也**不等于已经作用到现网**。

---

## 四、什么时候才允许动现网版

只有当用户明确表达类似意思时，才允许动现网：

- 上线到 PTRSS
- 同步到现网
- 把这版应用到正在跑的 PTRSS
- 重启 PTRSS 让修改生效

如果用户没有明确说这些，就默认：

- 不进容器
- 不 `docker cp`
- 不重启 `ptrss`
- 不覆盖现有 `/docker/PTRSS/data` 中的真实配置

---

## 五、现网应用的安全路径

当用户明确要求把改动上线到现网时，建议走：

1. 先完成仓库版修改
2. 在宿主机确认目标源码
3. 同步到运行容器
4. 重启 `ptrss`
5. 验证 Web / Telegram 实际表现

注意：

- 不能只看宿主机文件是否变了
- 必须验证容器里真正运行的代码
- 必须验证页面或 TG 实际结果

---

## 六、敏感信息规则

以下内容默认不得进入仓库：

- 真实 PT 站点 Cookie
- 真实 RSS passkey
- 真实 Telegram Bot Token / Chat ID
- 真实下载器地址 / 用户名 / 密码
- `data/web_auth.json`
- `data/web_session_secret.txt`
- `data/state.json`
- `data/ptrss.log`
- `data/tg_update_offset.txt`
- 任何运行态日志、缓存、状态文件

如果需要示例配置，应使用占位值。

---

## 七、建议的日常口令

为了避免误会，后面可以这样说：

### 只同步仓库
- 顺手同步 GitHub
- 只整理仓库
- 先别动现网

### 允许上线现网
- 上线到 PTRSS
- 把这版同步到现网
- 重启 PTRSS 让它生效

---

## 八、最短结论

**仓库不是现网。**  
**默认只改仓库，明确授权后才动现网。**
